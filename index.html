1. Project Objectives

Goal

Maximize the Longest Common Subsequence (LCS) score between retrieved document chunks and ground-truth evidence in a multi-domain RAG pipeline.

Challenge

Standard BM25 retrieval struggles with "noisy" chunks, diverse content types (tables, formulas), and the semantic gap between queries and documents.

SUCCESSFUL: +3.5% LCS

Teammate Report: Nacer

Topic: Document Re-Ranking Classifier

The Approach

Nacer implemented a two-stage retrieval pipeline. While BM25 is fast, it lacks semantic understanding. This approach uses BM25 to fetch candidates, then re-orders them using a Machine Learning model.

  • Stage 1: BM25 fetches top-5 candidates per query.
  • Stage 2: A Random Forest Classifier (200 trees) scores candidates based on 26 features.

Key Features Engineered

1. Structural: chunk_len, unique_terms
2. Similarity: jaccard_sim, semantic_bert_score
3. Context: window_occurrences, term_proximity
4. Statistical: sum_idf, avg_idf

Results

Verdict: The classifier successfully bridged the semantic gap, improving LCS from 80.34% to 83.13%.

Teammate Report: Sarah

Topic: Quality-Based Clustering & Enhancement

The Approach

Sarah treated the chunks as data points in a 15-dimensional space. Using K-Means Clustering, she identified 5 distinct "Quality Tiers".

The analysis revealed that the POOR tier (Cluster 2) consisted mainly of broken tables and mathematical formulas corrupted by OCR.

Enhancement Strategy

Instead of filtering these out, the team applied Targeted Enhancement:

  • Tables: Restructured using row/column heuristics.
  • Formulas: Converted corrupted text back to LaTeX format.

Results

By fixing only the "POOR" chunks (n=1,468), the system saw a massive jump in that specific cluster's performance (+27.37%) and a solid global gain.

UNSUCCESSFUL STRATEGY

Teammate Report: Asmaa

Topic: Syntactic Quality Filtering

Hypothesis: Removing "noisy" chunks (those with high ratios of non-alphanumeric characters) will improve retrieval precision by reducing false positives.

Methodology

Asmaa defined a "Noise Ratio" for every chunk:

Noise = (NonAlphaChars) / (TotalLength)

Chunks exceeding a threshold (0.4) were removed from the index entirely.

Findings

The experiment failed to improve results. The filtered dataset performed worse (LCS dropped from 0.85 to 0.72 in experiments).

Why? Technical documents contain high-noise content (code, JSON, formulas) that is actually relevant. "Noise" in technical docs is often "Signal".

DIAGNOSTIC TOOL

Teammate Report: Oumnia

Topic: Association Rules Analysis

Goal: Use data mining (Apriori/FP-Growth) to find hidden rules that predict when a chunk will be useful.

Key Rule Discovered:
{Short_Length, Low_Density, High_Word_Overlap} => {High_LCS_Score}

This method didn't directly improve the system but acted as a diagnostic tool, confirming that simple keyword matching (BM25) was the dominant factor in current success, and that re-ranking was necessary to capture longer, denser chunks.

4. Final Conclusions & Insights

What Worked

  • ✅ Supervised Re-Ranking: The most effective lever. Learning the relationship between query structure and document structure (+3.5%).
  • ✅ Targeted Repair: Fixing broken data (Sarah's method) is better than deleting it.
  • ✅ Semantic Features: Features like "Answer Likeness" and "IDF Sum" were high predictors of quality.

What Didn't Work

  • ❌ Aggressive Filtering: Asmaa's report proved that in technical domains, "ugly" data (symbols, code) is often the answer. Deleting it hurts Recall.
  • ❌ Pure Syntactic Rules: Oumnia's analysis showed that simple rules (length, density) aren't enough to capture complex relevance.
